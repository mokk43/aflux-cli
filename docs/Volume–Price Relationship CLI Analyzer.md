# Volume–Price Relationship CLI Analyzer
## Problem statement
Add a Python CLI command that accepts more than five days of one stock’s historical closing prices and whole-day transaction volumes, then classifies the observed volume–price relationship using the patterns described in `docs/volume–price-relationship.md`.
## Current state
The existing CLI entry point is `aflux/cli.py`, built with Typer and exposed as the `aflux` command through `pyproject.toml`. Existing scan output supports table, JSON, and CSV patterns through `aflux/output.py`. Shared data models live in `aflux/models.py`, and pure market calculation logic belongs outside `cli.py`, matching the repository rule that CLI code should parse input and render output only.
The design doc defines qualitative patterns such as 放量上涨, 缩量上涨, 放量滞涨, 放量下跌, 缩量下跌, 底部放量, 天量天价, 地量地价, 顶背离, 底背离, and 横盘 variants, but it does not define numeric thresholds. The implementation therefore needs explicit defaults and user-configurable thresholds.
## Recommended approach
Implement a standalone local-data analyzer as `python aflux/volume_price.py INPUT`, where `INPUT` is a CSV file containing at least `date`, `close`, and `volume` fields for a single stock. This avoids network dependencies, matches the preferred invocation style, and keeps the first version easy to run without changing the packaged `aflux` command surface.
Keep `aflux/volume_price.py` structured so the classification logic is still pure and testable, with the script entry point limited to argument parsing, standard-library CSV loading, and rendering. Use dataclasses or internal typed structures in the standalone module instead of adding Pydantic models or using pandas for the first version, because the input shape is simple and local.
## Classification design
Normalize input rows by date ascending, require at least six valid rows, and fail fast on invalid required data: missing required columns, dates not in `YYYY-MM-DD` format, duplicate dates, non-numeric close or volume, close <= 0, or volume <= 0. Date validation should produce row-specific errors. Require `lookback >= 5` and `len(rows) >= lookback + 1`; for the default `--lookback 5`, this means at least six rows, and `--lookback 10` requires at least eleven rows. For v1, classify only the latest state using all provided recent data and the configured lookback window; do not emit per-day classifications.
Use exact metric formulas. `previous_close` is the close immediately before the latest row. `baseline_close` is the close immediately before the prior-lookback window when present; for the minimum valid case, it is the first row in the prior-lookback window. `latest_price_change_pct = (latest_close - previous_close) / previous_close * 100`. `recent_price_change_pct = (latest_close - first_prior_lookback_close) / first_prior_lookback_close * 100`. `prior_avg_volume = avg(prior_lookback_volumes)`. `latest_volume_ratio = latest_volume / prior_avg_volume`. `sideways_range_pct = (max(prior_lookback_and_latest_closes) - min(prior_lookback_and_latest_closes)) / min(prior_lookback_and_latest_closes) * 100`. For output precision, render percentages and ratios with 1 decimal place in table output, JSON output, CSV output, and exports.
Classify trends from those formulas. Rising trend is `recent_price_change_pct > price_flat_pct`; falling trend is `recent_price_change_pct < -price_flat_pct`; otherwise price trend is flat. Volume expansion is `latest_volume_ratio >= volume_expand_ratio`; volume shrinkage is `latest_volume_ratio <= volume_shrink_ratio`; otherwise volume is neutral. Sideways is `sideways_range_pct <= sideways_range_pct_threshold`. Use the full CSV history, not only the lookback window, to decide whether the latest close is in a high or low area; report this basis explicitly as `position_range: full_input`. Set `position_state` to `high_area` when `latest_close >= full_input_high * (1 - position_threshold_pct / 100)`, `low_area` when `latest_close <= full_input_low * (1 + position_threshold_pct / 100)`, and `middle_area` otherwise. If both high and low conditions are true in a very narrow full-input range, prefer `middle_area` unless the latest close is also an absolute max or min handled by higher-priority rules.
Use rule-based classification in explicit priority order so conflicts are deterministic. Ties count as records: `latest_close == full_input_high` counts as full-input max close even if earlier rows share the same close; `latest_close == full_input_low` counts as full-input min close; `latest_volume == full_input_max_volume` counts as full-input max volume; and `latest_volume == full_input_min_volume` counts as full-input min volume. First, if latest close is the full-input max close and latest volume is the full-input max volume, classify `天量天价`. Second, if latest close is the full-input min close and latest volume is the full-input min volume, classify `地量地价`. Third, if latest close is the full-input max close and latest volume is not the full-input max volume, classify `顶背离`. Fourth, if latest close is the full-input min close and latest volume is not the full-input min volume, classify `底背离`. Fifth, evaluate sideways and breakout patterns. Sixth, evaluate trend-specific pairings such as 量增价涨, 量缩价涨, 放量滞涨, 放量下跌, 缩量下跌, and 底部放量. For 天量/天价/地量/地价 and divergence, require absolute max/min records within the full CSV input, not percentile near-record status.
Define breakout and bottom-volume rules exactly. `upward_breakout` means latest close is greater than `max(prior_lookback_closes)` and `latest_volume_ratio >= volume_expand_ratio`. `downward_breakdown` means latest close is less than `min(prior_lookback_closes)` and `latest_volume_ratio >= volume_expand_ratio`. `底部放量` requires `position_state == low_area`, recent price trend is falling or flat, latest-day price is flat or rising versus previous close, latest volume is expanded, and latest close is not the full-input min close because full-input-min cases are handled by higher-priority `地量地价` or `底背离`. Latest-day price is flat or rising when `latest_price_change_pct >= -price_flat_pct`; with the selected default this means a latest-day decline up to 2% still counts as not materially falling.
Return one `primary_type` plus a `signals` list for overlapping evidence, supporting metrics, and neutral Chinese `interpretation` and `risk_note` fields. Do not output direct buy/sell/hold action labels from the doc as primary guidance, and do not print a separate disclaimer banner; keep cautionary language contextual inside `risk_note`. If evidence is weak or mixed, return `primary_type` as `不明确` with low confidence instead of forcing a bullish or bearish label. Confidence is `high` when a primary rule matches with at least two supporting signals, `medium` when a primary rule matches with one supporting signal or mixed context, and `low` for `不明确` or weak/conflicting signals.
Default thresholds should be configurable through CLI options. Selected defaults: volume expansion at current volume >= 1.5 times prior-lookback average volume, volume shrinkage at current volume <= 0.8 times prior-lookback average volume, price flat when absolute latest or recent price change <= 2%, sideways range when recent high-low range <= 3%, high/low position when latest close is within 5% of the full-input high/low, and breakout volume confirmation at >= 1.5 times the prior-lookback average volume.
## CLI and output
Add a script interface: `python aflux/volume_price.py data.csv`. The CSV must include exact English headers `date`, `close`, and `volume`; optional exact English `code` and `name` columns improve output labels but are not required. Do not support Chinese or provider-specific column aliases in v1. Allow extra CSV columns and ignore them. Treat `volume` as unit-agnostic activity data: shares, lots, or transaction amount are all acceptable as long as every row uses the same unit. Default output to `table`, with explicit `--format json` and `--format csv` for automation. Support `--export` with export format inferred from `.json` or `.csv`; unsupported export extensions fail fast. Console `--format` controls stdout only, so users can view a table while exporting machine-readable output. Support threshold overrides such as `--volume-expand-ratio`, `--volume-shrink-ratio`, `--price-flat-pct`, `--sideways-range-pct`, `--position-threshold-pct`, and `--lookback`. Keep diagnostics on stderr and data output on stdout.
Use standard-library-only rendering for this standalone script. Do not require Rich or pandas inside `aflux/volume_price.py`; plain aligned text is enough for table output, and `json`/`csv` from the standard library should handle machine-readable output. Exit with code `0` on success, `1` for validation/input/export errors, and `2` for unexpected runtime errors.
Define a stable result schema shared by table, JSON, CSV, and export output. Top-level fields: `code`, `name`, `date_range`, `latest_date`, `primary_type`, `signals`, `confidence`, `metrics`, `interpretation`, and `risk_note`. The `date_range` object contains `start` and `end`. The `metrics` object contains `latest_close`, `latest_volume`, `previous_close`, `latest_price_change_pct`, `recent_price_change_pct`, `prior_avg_volume`, `latest_volume_ratio`, `sideways_range_pct`, `position_range`, `position_state`, `full_input_high`, and `full_input_low`.
The table output should show the stock code/name when present, date range, primary relationship type, secondary signals, confidence level, latest close, latest volume, latest price change, latest volume ratio, recent price change, sideways range, position state, neutral interpretation, and risk note. Confidence should be categorical: `high`, `medium`, or `low`, not a decimal score, because this is a rule-based classifier rather than a statistical model. JSON/CSV should be machine-readable. CSV should use this exact column order: `code`, `name`, `date_range_start`, `date_range_end`, `latest_date`, `primary_type`, `signals`, `confidence`, `metric_latest_close`, `metric_latest_volume`, `metric_previous_close`, `metric_latest_price_change_pct`, `metric_recent_price_change_pct`, `metric_prior_avg_volume`, `metric_latest_volume_ratio`, `metric_sideways_range_pct`, `metric_position_range`, `metric_position_state`, `metric_full_input_high`, `metric_full_input_low`, `interpretation`, `risk_note`. CSV should flatten nested fields, and `signals` should be semicolon-separated text.
## Testing and validation
Add automated unit tests for the pure classifier using Python’s built-in `unittest`, avoiding new test dependencies for v1. Synthetic sequences should cover the major pattern families from the doc: 放量上涨, 缩量上涨, 放量滞涨, 放量下跌, 缩量下跌, 底部放量, 天量天价, 地量地价, 顶背离, 底背离, 缩量横盘, 放量横盘, and breakout confirmation. Add validation tests for missing columns, duplicate dates, bad date format, non-numeric close or volume, close <= 0, volume <= 0, `lookback < 5`, `lookback >= len(rows)`, unsupported export extension, and extra columns being ignored. Run tests with `python -m unittest`. Also validate the standalone CLI manually with small fixture files, add `examples/volume_price_sample.csv` as a runnable fixture, and run `ruff check aflux/` and `ruff format --check aflux/`.
## Options considered
Option 1 is a local-file CLI command inside the existing `aflux` executable. Effort is moderate, risk is low, and it builds on the current Typer/Pydantic/Rich structure, but it changes the packaged command surface.
Option 2 is a standalone script: `python aflux/volume_price.py data.csv`. Effort is lower, it matches the requested invocation, and it is the selected option for the first version. The main tradeoff is that it bypasses the existing Typer command conventions, so the module should be written cleanly enough to promote into `aflux` later if needed.
Option 3 is an API/server feature. Effort and surface area are higher, and it is not justified until the local classifier semantics are proven.
## Parallelization
Parallel child agents are not useful for the first implementation plan because the work is tightly coupled: the threshold semantics, model shape, CLI interface, renderer, and tests all depend on the same classification contract. A single implementation path will avoid merge churn and inconsistent rule definitions.
## Rollback and dependencies
No external API, account, token, or third-party service is required. Rollback is simple because this adds an isolated script module, tests, and a small example CSV without changing existing scan behavior, CLI commands, or cache data.
